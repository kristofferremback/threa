# Workspace Research: Streaming Feedback, Wall-Clock Bounds, and Graceful Abort

## Context

During a demo the workspace research path took ~30 minutes to complete with zero user feedback beyond a static "Searching workspace…" label. Two failure modes are layered: (1) the research loop is unbounded in wall-clock time and silently hangs on slow OpenRouter providers, and (2) the user has no visibility into _what_ research is doing or _how_ to stop it. When the workspace is large (thousands of messages) the failure mode bites hardest.

This is the GAM (General Agentic Memory, arxiv 2511.18423) deep-research path. It is the product's core differentiator and must not be deleted. The fix is to keep the GAM plan→execute→evaluate→iterate loop intact while:

- Capping wall-clock time and per-call latency.
- Streaming substep text to the user as research progresses.
- Letting the user click **Stop research** to abort gracefully — research returns whatever partial context was found, the companion agent continues normally, the session does NOT die.
- Tightening the planner/evaluator (model swap to Haiku 4.5, iteration cap 5→2, evaluator-bias toward `sufficient`, parallel speculative baseline+plan on iteration 1, stricter system prompts).

Memorizer-side improvements (some memos in the wild are garbage) and session-scoped research memoization are deferred to separate PRs.

## Architecture decisions (after Plan-agent review)

- **Substep streaming uses a NEW dedicated event** `agent_session:substep` (not an extension of `agent_session:progress`). The progress event fires on every step type and overloading it would cause cross-feature drift. The new event is additive — old clients ignore it.
- **Substeps are partially ephemeral, fully persisted on completion.** During step execution they stream live via `agent_session:substep` events (smooth UX, no latency). Simultaneously the researcher accumulates a `substeps: [{ text, at }]` log. On step completion the log is baked into the existing `step.content` JSON via the tool's `trace.formatContent`, so it persists to DB through the existing `ActiveStep.complete({ content })` path — no new table, no migration, no new persistence code. On refresh, the trace dialog bootstrap fetches the persisted step and renders the full substep history. In-flight steps continue to receive live socket events after refresh/rejoin, so no state gap.
- **Frontend "wipe ephemeral on completion" semantics:** when a step completes (`agent_session:step:completed`), the frontend clears the in-memory ephemeral substep state for that step, and the persisted version from the step record takes over. The timeline card goes back to its completed-state rendering; the trace dialog shows the full substep list from `step.content`.
- **Observer is NOT restructured to create steps on `tool:start`.** Instead a new `tool:progress` AgentEvent variant is added; the observer translates it to `SessionTrace.emitSubstep()` (ephemeral socket emission, no DB write). The DB write still happens at `tool:complete` through the normal content path. This is a much smaller blast radius than mutating the persisted step lifecycle.
- **Abort registry is generic (`SessionAbortRegistry`), not research-specific.** Same shape will support future long-running tools (web_search, read_url of large docs).
- **Graceful abort never touches `AgentRuntime.shouldAbort`** — that mechanism throws and fails the session. Graceful abort sets a `SessionAbortRegistry`-owned `AbortController`; the researcher checks it at safe checkpoints, returns `{ partial: true, partialReason: "user_abort", … }`, and the agent loop continues.
- **Per-call signal composition.** The researcher composes a per-call AbortSignal from (a) the user-abort signal, (b) a total-deadline timer, (c) a per-call timeout. AI calls that throw `AbortError` are caught and treated as a soft "no result" so the loop can continue or partial-return cleanly.
- **Cost recording is automatically skipped on aborted AI calls** because the `await` throws and `maybeRecordUsage` is never reached. Verified — no extra work needed.
- **Defaults for the open UX questions** (no user input asked because the simpler choice fits the spec): partial results are signalled in the tool result JSON (`{ partial: true, partialReason }`), the LLM is told in the system prompt how to handle them; when aborted, the timeline card stays in "running" state until the persona's normal response/completion arrives; abort button is wired to `workspace_research` only in V1 (the registry stays generic so it's trivial to wire to other tools later).

## Out of scope (explicit non-goals)

- Memorizer quality investment (Option H) — separate PR; user wants to focus on it properly.
- Session-scoped research memoization (Option I) — separate PR.
- Aborting non-research tools — registry is generic enough to support it later, but `toolSignalProvider` matches only `workspace_research` in V1.
- Touching `AgentRuntime.shouldAbort` semantics — fatal session abort remains for deleted/superseded.
- **New DB columns or migrations for substeps** — substeps are stored inside the existing `step.content` JSON blob. Not a schema change.
- Adding a "partial context" status badge to the persona's final message in the UI — JSON-only signalling via tool result; the LLM handles user-facing wording.
